# Agent Stream Tail Glow Design

**Goal:** 将 Agent 流式回复的视觉反馈改成“仅高亮最新增长边界”的样式：没有文本时继续显示三个点；一旦开始出字，历史文本保持完全静态，只有最新增长出来的那一小段文字带一次短暂的流光。

## Problem

当前 Agent 的等待态有两个问题：

- 没有文本时使用三个点，这部分是可接受的。
- 一旦文本开始出现，当前实现仍然会在文本下方继续显示等待指示器，视觉上显得重复且廉价。

用户想要的效果更接近元宝：

- 已经出现的文本不动、不闪、不变色。
- 只在“刚刚长出来的文本边缘”给一个很淡的流光过渡，暗示“还没输完”。

## Approach

不做整段文本流光，也不对整个气泡做动画；只追踪**最近一次进入 UI 的流式增量**。

在 Agent 本地事件状态中为流式 `text` 事件增加两段本地元数据：

- `streamTail`: 最近一次进入 UI 的文本增量
- `streamPulseKey`: 每次增量更新时递增，用于重启动画

渲染时：

- `content - streamTail` 视为稳定文本
- `streamTail` 视为最新增长边界

## Rendering Rules

### 1. 没有文本时

- 继续显示三个点
- 不改现有占位逻辑

### 2. 有文本且仍在 streaming 时

- 历史文本：普通静态文本，无动画
- 最新 `streamTail`：一段很短的淡扫光动画，持续一次后停住
- 不再显示三个点

### 3. streaming 结束后

- 整条消息恢复为标准 `MarkdownMessage`
- 不保留任何动画感

## Tradeoff

为了把“历史文本”和“最新尾部文本”明确拆开，流式中的这条 Agent 消息会在 streaming 阶段先按普通文本样式渲染，而不是实时走完整 Markdown 渲染；消息完成后再恢复为标准 Markdown 展示。

这是有意的折中：

- 优点：可以精准控制只有最新边界发光
- 代价：在极少数复杂 Markdown 场景下，流式期间的即时格式感会弱一点，但完成后会恢复正确格式

## UX Constraints

- 动画必须轻，不能闪烁
- 只在 Agent 页面生效，不影响 Chat
- 遵守 `prefers-reduced-motion`
- 保持文本可读性，不能牺牲对比度

## Verification

- 没文本时：三个点不变
- 开始出字后：三个点消失，只有最后增长出来的那几字有淡流光
- 已出现文字完全静态
- 回复完成后：整条消息恢复普通 Markdown 样式
